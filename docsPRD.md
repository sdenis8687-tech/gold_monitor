# PRD — Gold Monitor

## 1. Продукт
Gold Monitor — внутренний web-инструмент мониторинга стоимости золота в рублях для заказчика.

## 2. Цель
Заменить n8n workflow отдельным продуктом `web + api + worker`, который:
- получает частые обновления цены золота и USD/RUB;
- показывает историю на одной странице;
- позволяет визуально сравнивать золото и доллар;
- отправляет email-уведомления по осмысленным порогам.

## 3. Scope
В scope входят:
- внутренняя dashboard-страница;
- история цен;
- сравнение золота и USD/RUB;
- email alerts;
- хранение raw и derived данных;
- Docker deployment.

В scope не входят:
- публичный data API для третьих лиц;
- перепродажа рыночных данных;
- личный кабинет с ролями;
- админка с управлением пользователями;
- мобильное приложение.

## 4. Источник данных
Production source:
- `MOEX ISS`

Instruments:
- `GLDRUB_TOM`
- `USD000UTSTOM`

## 5. Бизнес-логика
Использовать такие формулы:
- `gold_999_rub_g = last_price(GLDRUB_TOM)`
- `gold_585_rub_g = round(gold_999_rub_g * 585 / 999, 2)`
- `usd_rub = last_price(USD000UTSTOM)`

## 6. Наследуемая логика из legacy
Переносим:
- периодический сбор данных;
- формулу 585 через 999;
- хранение истории;
- сравнение с предыдущим значением;
- email-уведомления;
- исторический обзор.

Не переносим:
- 1000bankov как источник;
- QuickChart email как основной интерфейс;
- n8n как runtime.

## 7. Главная страница
На странице должны быть:
- KPI `Золото 585, ₽/г`
- KPI `Золото 999, ₽/г`
- KPI `USD/RUB`
- KPI `Последнее обновление`

Под KPI должны быть:
- график `Золото 585, ₽/г`
- график `USD/RUB`

Ниже должна быть:
- кнопка/переключатель `Таблица`
- табличный режим

## 8. Графики
Обязательные требования:
- график золота показывает только `585`
- `999` не выводится как отдельная серия
- оба графика синхронизированы по hover/crosshair
- период единый для обоих графиков
- периоды:
  - `1D`
  - `7D`
  - `30D`
  - `90D`
  - `1Y`

## 9. Режим нормализации
На странице обязательно есть переключатель:
- `Абсолютные значения`
- `Нормализовать к 100`

Формула нормализации:
- `normalized_value = round(current_value / first_visible_value * 100, 2)`

## 10. Табличный режим
Кнопка `Таблица` переключает страницу в табличный режим без потери выбранного периода.

Таблица обязана содержать колонки:
- `Дата/время`
- `Золото 585, ₽/г`
- `Золото 999, ₽/г`
- `USD/RUB`

Правила таблицы:
- сортировка по времени по убыванию по умолчанию;
- одинаковый range с графиками;
- данные берутся из того же API, что и графики;
- пагинация по 100 строк.

## 11. Статусы интерфейса
Нужно реализовать:
- `loading`
- `ready`
- `stale`
- `empty`
- `error`

Правило stale:
- если возраст последней source timestamp > 25 минут, данные считаются stale;
- stale данные показываются, но с явной визуальной пометкой.

## 12. Частота обновления
- worker опрашивает источник каждые 5 минут;
- UI обновляет latest snapshot каждые 60 секунд через API приложения;
- UI не опрашивает MOEX напрямую.

## 13. Уведомления
Email alerts поддерживают:
- абсолютный порог;
- процентный порог;
- cooldown;
- daily digest.

MVP-мониторинг для алертов:
- основная метрика: `gold_585_rub_g`

## 14. Архитектура
- Frontend: Next.js
- API: Fastify
- Worker: Node service
- DB: PostgreSQL
- ORM: Prisma
- Deploy: Docker Compose

## 15. Критерии приемки
Продукт считается готовым, когда:
- построен без n8n в runtime;
- source of truth = `MOEX ISS`;
- worker сохраняет raw и derived data;
- API отдает current + history;
- UI показывает KPI;
- UI показывает два графика;
- на золотом графике только `585`;
- кнопка `Таблица` работает;
- email alerts работают;
- тесты проходят;
- verification report создан.
